# 智能客服 · 大模型选型与部署方案

> 目标:为本平台「智能客服」选择大模型并给出**最低成本(API)** 与 **最低成本(自建/离线)** 两套落地方案,
> 且**与现有前后端实现严格匹配,零代码改动**。

---

## 0. 现有技术约束(决定选型边界)

后端 `OpenAiCompatProvider`(`com.codeman.platform.cs.service`)的事实:

| 约束 | 说明 | 选型含义 |
|---|---|---|
| 统一走 **OpenAI 兼容接口** | `POST {base-url}/chat/completions`,`stream:true`,`Authorization: Bearer {key}` | 任何 OpenAI 兼容端点(云 API / 网关 / 本地)即插即用 |
| **SSE 流式** | 读取上游 `data: {choices[].delta.content}` 转发前端 | 模型/服务需支持 stream(主流全支持) |
| **关键词 RAG**(2-gram) | 知识库召回在本地,不依赖向量/嵌入模型 | **无需** embedding 模型,小模型即可胜任 |
| **配置驱动 + 可降级** | `app.ai.*`;未配置时回退"知识库匹配" | 切换模型=改 3 个环境变量 |
| **密钥 env/Nacos/KMS 注入** | `AI_API_KEY` 不落明文 | API/自建均适用 |

> 结论:智能客服是**「FAQ/产品问答 + 知识库参考」**型负载,**不需要顶级推理模型**;
> 选"便宜、够用、OpenAI 兼容"的中小模型即可,重点在**成本**与**数据合规**。

**对接只改这三项(任何方案通用):**
```bash
AI_ENABLED=true
AI_BASE_URL=<端点>      # 末尾通常带 /v1;代码会自动拼 /chat/completions
AI_API_KEY=<密钥>       # 本地模型可留空
AI_MODEL=<模型名>
# 可选:AI_TIMEOUT_SEC、AI_HISTORY_LIMIT
```

## 1. 推荐结论(先看这里)

| 场景 | 推荐 | 月成本量级 | 数据出网 |
|---|---|---|---|
| **数据可出网、要最省心最省钱** | 云 API:**智谱 GLM-4-Flash(常有免费额度)** 或 **DeepSeek-V3 / 通义 qwen-turbo** | 0 ~ 几十元 | 是 |
| **数据须内网/合规、单机即可** | 自建 **Ollama + Qwen2.5-7B-Instruct(Q4)**,单张 12–24G 显卡 | 仅电费/硬件摊销 | **否(完全离线)** |
| **内网 + 高并发生产** | **vLLM + Qwen2.5-14B-Instruct(AWQ)**,24–48G 显卡 | 硬件+电费 | **否** |
| **完全无 GPU 的小流量内网** | Ollama + **Qwen2.5-3B / 1.5B-Instruct**(CPU 也能跑) | 仅电费 | **否** |

> 一句话:**能出网就用 GLM-4-Flash/DeepSeek/qwen-turbo(近乎免费);要合规就 Ollama 跑 Qwen2.5-7B。**

## 2. 方案 A:云端 API(最低成本、最省心)

适合:对数据出网无硬性限制。优点:零运维、随用随付、低流量近乎免费。

### 2.1 候选与价格(人民币 / 每百万 token,约值,**以各家官网为准**)

| 厂商·模型 | 输入 | 输出 | 备注 |
|---|---|---|---|
| **智谱 GLM-4-Flash** | 0(常驻免费)/极低 | 0 / 极低 | 客服首选,免费额度大;`glm-4-flash` |
| **通义 qwen-turbo** | ~¥0.3 | ~¥0.6 | 阿里云百炼,极便宜、稳定;`qwen-turbo` |
| **通义 qwen-plus** | ~¥0.8 | ~¥2 | 质量更好一档;`qwen-plus` |
| **DeepSeek-V3(deepseek-chat)** | ~¥2(命中缓存更低) | ~¥8 | 性价比高,夜间半价;`deepseek-chat` |
| **百度 ERNIE-Speed/Lite** | 多为免费/极低 | 免费/极低 | 文心轻量版 |
| **Kimi(moonshot-v1-8k)** | ~¥12 | ~¥12 | 长文本强,单价偏高 |
| **OpenAI gpt-4o-mini** | ~$0.15 | ~$0.6 | 需海外网络;`gpt-4o-mini` |

> 客服问答**不要用** R1/o 系列等"推理(reasoning)"模型——更贵更慢,FAQ 用不上。

### 2.2 月成本估算(典型客服量)

假设:2,000 次会话/月 × 4 轮 × 每次约 1.5K 输入 + 0.3K 输出 ≈ **输入 12M / 输出 2.4M tokens/月**。

- qwen-turbo:≈ 12×0.3 + 2.4×0.6 ≈ **¥5 / 月**
- DeepSeek-V3:≈ 12×2 + 2.4×8 ≈ **¥43 / 月**(夜间/缓存更低)
- GLM-4-Flash:**¥0(免费额度内)**

> 低流量下 API 成本可忽略,**比自建省心得多**;只有在高并发或合规要求时才需要自建。

### 2.3 配置示例(改环境变量即可)

```bash
# 智谱 GLM-4-Flash
AI_ENABLED=true AI_BASE_URL=https://open.bigmodel.cn/api/paas/v4 AI_API_KEY=*** AI_MODEL=glm-4-flash

# 通义 qwen-turbo(阿里云百炼,OpenAI 兼容模式)
AI_ENABLED=true AI_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1 AI_API_KEY=*** AI_MODEL=qwen-turbo

# DeepSeek
AI_ENABLED=true AI_BASE_URL=https://api.deepseek.com/v1 AI_API_KEY=*** AI_MODEL=deepseek-chat

# OpenAI
AI_ENABLED=true AI_BASE_URL=https://api.openai.com/v1 AI_API_KEY=*** AI_MODEL=gpt-4o-mini
```

> 多家随时切换/灰度:可加一层 **one-api / new-api** 网关(也 OpenAI 兼容),`AI_BASE_URL` 指网关即可统一计费与限流。

## 3. 方案 B:自建 / 离线(数据不出网、最低边际成本)

适合:数据合规、内网隔离。一次性硬件/电费,边际成本≈0;**完全不出公网**。

### 3.1 模型档位 × 硬件(开源、商用友好,OpenAI 兼容服务暴露)

| 硬件 | 推荐模型 | 量化/显存 | 适用 |
|---|---|---|---|
| **无 GPU(CPU)** | Qwen2.5-3B-Instruct / 1.5B | GGUF Q4 ≈ 2–3GB 内存 | 低频内网 FAQ,几 token/s |
| 1× 8–12GB GPU | **Qwen2.5-7B-Instruct** | Q4_K_M ≈ 5GB 显存 | 单租户/中小并发(**甜点**) |
| 1× 16–24GB(3090/4090) | Qwen2.5-14B-Instruct | Q4 ≈ 9GB / 全精度 7B | 质量更好,可中并发 |
| 24–48GB | Qwen2.5-32B-Instruct | AWQ ≈ 20GB | 高质量内网 |
| 多卡/生产 | 14B/32B + **vLLM** | 连续批处理 | **高并发**(几十~上百并发) |

> 同级备选:GLM-4-9B-Chat、Llama-3.1-8B-Instruct、Qwen3-8B/14B。中文客服优先 **Qwen2.5 系列**。

### 3.2 最简:Ollama(开发/单机首选)

```bash
curl -fsSL https://ollama.com/install.sh | sh
ollama pull qwen2.5:7b-instruct        # 无 GPU 用 qwen2.5:3b-instruct
ollama serve                            # 监听 :11434,自带 OpenAI 兼容 /v1
```
对接(无需 key):
```bash
AI_ENABLED=true AI_BASE_URL=http://127.0.0.1:11434/v1 AI_MODEL=qwen2.5:7b-instruct
```

### 3.3 生产高并发:vLLM(吞吐最佳,OpenAI 兼容)

```bash
pip install vllm
python -m vllm.entrypoints.openai.api_server \
  --model Qwen/Qwen2.5-7B-Instruct --port 8000 --max-model-len 8192
# 显存紧张:--quantization awq 用 Qwen/Qwen2.5-14B-Instruct-AWQ
```
对接:
```bash
AI_ENABLED=true AI_BASE_URL=http://<vllm-host>:8000/v1 AI_MODEL=Qwen/Qwen2.5-7B-Instruct
```

### 3.4 docker-compose 集成 Ollama(与本项目并跑)

```yaml
services:
  ollama:
    image: ollama/ollama:latest
    ports: ["11434:11434"]
    volumes: ["ollama:/root/.ollama"]
    # GPU: 取消注释
    # deploy: { resources: { reservations: { devices: [{ capabilities: ["gpu"] }] } } }
  backend:
    environment:
      AI_ENABLED: "true"
      AI_BASE_URL: "http://ollama:11434/v1"
      AI_MODEL: "qwen2.5:7b-instruct"
volumes: { ollama: {} }
```
> 首次需 `docker exec -it <ollama> ollama pull qwen2.5:7b-instruct`。

## 4. API vs 自建 怎么选(盈亏与合规)

- **流量小 / 可出网** → 选 **API(GLM-4-Flash 免费 或 qwen-turbo)**:月成本 0~几十元,免运维。
- **数据须内网 / 合规** → 选 **Ollama 7B**:一次性显卡(消费级即可)+ 电费,数据零出网。
- **高并发内网** → **vLLM 14B**:吞吐与质量兼顾。
- 盈亏平衡:仅当**调用量极大**或**合规/隔离**成为硬约束时,自建才比 API 更划算;否则 API 更省。

## 5. 与现有实现的对接清单(均零代码改动)

1. **base-url 末尾**:云 API 用 `.../v1`;Ollama `http://host:11434/v1`;vLLM `http://host:8000/v1`(代码自动拼 `/chat/completions`)。
2. **AI_MODEL** 用各端点的模型名(如 `glm-4-flash`、`qwen-turbo`、`qwen2.5:7b-instruct`)。
3. **密钥**:云 API 经 env/Nacos/KMS 注入;本地模型留空。
4. **超时/历史**:`AI_TIMEOUT_SEC`(本地小模型可调大)、`AI_HISTORY_LIMIT`(控成本)。
5. **降级**:未配置或上游故障 → 自动回退知识库匹配,客服窗不中断。
6. **知识库(可选增强)**:当前为关键词召回,**够用**;若将来要语义检索,可用 `bge-m3`(Ollama `ollama pull bge-m3`)做 embedding,仅替换 `KnowledgeBase` 实现,上层无感。
7. **护栏不变**:system 提示已限定"仅答本产品、不外泄密钥/隐私、不写操作"。

## 6.5 落地默认值与一键联调(已写入仓库)

已把默认配置写入仓库,**开箱即用、按需开启**:

**application.yml(默认指向本地 Ollama,仍由开关控制)**
```yaml
app:
  ai:
    enabled:   ${AI_ENABLED:false}                       # 默认关闭→走知识库降级,不报错
    base-url:  ${AI_BASE_URL:http://127.0.0.1:11434/v1}  # 默认本地 Ollama;云 API 改此值
    api-key:   ${AI_API_KEY:}                            # 本地留空;云 API 注入
    model:     ${AI_MODEL:qwen2.5:7b-instruct}           # 云 API 改 glm-4-flash / qwen-turbo / deepseek-chat
```

**docker-compose.yml**:`backend` 已注入 `AI_*`(默认指向 `ollama` 服务);新增可选 `ollama` 服务(profile=`ai`)。

一键离线启用(自建路线):
```bash
docker compose --profile ai up -d                       # 起 ollama
docker compose exec ollama ollama pull qwen2.5:7b-instruct
AI_ENABLED=true docker compose up -d backend            # 打开开关(或写入 .env)
```
切云 API(出网路线,最省):在 `.env` 设 `AI_ENABLED=true`、`AI_BASE_URL`、`AI_API_KEY`、`AI_MODEL` 后 `up -d backend` 即可,**compose 与代码均不改**。

### 真机联调验证结果(本仓库已跑通)

由于受控环境无公网/GPU,使用零依赖的 **OpenAI 兼容模拟模型** `demo/ai-mock-server.py` 代替 Ollama/云 API
做端到端验证(**线路、协议、配置与真实端点完全一致**):

```bash
python3 demo/ai-mock-server.py 8770 &
AI_ENABLED=true AI_BASE_URL=http://127.0.0.1:8770/v1 AI_MODEL=demo-llm AI_API_KEY=demo \
  java -jar server/license-platform/target/license-platform-1.0.1.jar &
# /api/cs/status → {"ready":true,"model":"demo-llm","kbSize":12}
# /api/cs/chat   → 逐字 SSE(event:delta ...),且注入了知识库片段
```
实测 `status` 返回 `ready:true`,`chat` 返回逐字流式回复——证明**只改 3 个环境变量即接入**;
把 base-url/model 换成真实 Ollama(`http://ollama:11434/v1`,`qwen2.5:7b-instruct`)或云 API(如 `glm-4-flash`)即为生产形态。

## 7. 最终建议

- **默认推荐**:先用 **智谱 GLM-4-Flash**(免费/极低)或 **通义 qwen-turbo** 跑起来,成本几乎为零、质量满足客服。
- **有合规要求**:**Ollama + Qwen2.5-7B-Instruct**(单卡 12–24G),完全内网离线;并发上来再换 **vLLM + 14B-AWQ**。
- 无论哪种,**只改 3 个环境变量**,前后端与知识库/护栏/降级逻辑全部复用。
