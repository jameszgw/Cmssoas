03 · License 认证方案

目标：支持离线 / 在线两种授权方式；License 可管理、可续期、可变更、可吊销；具备版本号与完整历史追溯；防伪造、防篡改、防共享、防时间回拨。

1. 信任根：非对称签名

License 的安全基石是非对称密码学：

  • 厂商侧：持有私钥（在 KMS/HSM 中），用于对 License 内容签名。
  • 客户端 SDK：内置公钥，用于验签。
  • 结论：客户端即使被完全逆向，也拿不到私钥 → 无法伪造合法 License。攻击者只能尝试 patch 验签逻辑，而这由"代码保护层（02 文档）"对抗。

1.1 签名算法选型对比

算法签名长度性能安全性国密合规评价
RSA-2048长(256B)验签快、签名慢足够否兼容性最好，传统选择
ECDSA P-256短(~64B)快高否短签名，适合嵌入
Ed25519短(64B)最快、抗侧信道高否推荐（非国密场景）
SM2(国密)短快高是信创/政企推荐

结论：通用场景用 Ed25519；信创/政企合规场景用 SM2。License 内容采用 "claims(JSON) + 签名" 结构（类似 JWS 但自定义），离线为 .lic 文件，在线为 API 返回的签名快照。

2. License 数据模型（Claims）

{
  "schemaVersion": 2,              // License 结构版本号（升级兼容用）
  "licenseId": "LIC-2026-0001",   // 全局唯一
  "licenseVersion": 3,            // 该 License 的修订版本号（历史追溯）
  "tenantId": "T-10086",          // 所属租户
  "customer": "XX银行",            // 客户信息
  "productCode": "CODEMAN",       // 产品
  "edition": "ENTERPRISE",        // 版本/版型
  "mode": "OFFLINE",              // OFFLINE | ONLINE | HYBRID
  "issuedAt": "2026-06-06T10:00:00Z",
  "notBefore": "2026-06-06T00:00:00Z",
  "notAfter":  "2027-06-06T00:00:00Z",   // 有效期
  "machineCode": "MC-ab12cd34...",       // 硬件指纹绑定（离线必填）
  "machineBindPolicy": "STRICT",         // STRICT | LOOSE(允许部分硬件变更)
  "appVersionRange": ">=2.0.0 <3.0.0",   // 授权的软件版本号范围
  "modules": ["RISK", "REPORT", "AUDIT"],// 授权模块
  "features": {                          // 功能点 + 配额
      "RISK.REALTIME": true,
      "REPORT.EXPORT": true,
      "MAX_USERS": 200,
      "MAX_QPS": 500
  },
  "concurrency": 5,               // 在线浮动席位数
  "onlineCheck": {                // 在线模式参数
      "endpoint": "https://lic.vendor.com",
      "heartbeatSec": 300,
      "gracePeriodHours": 72      // 断网宽限期
  },
  "watermark": "WM-7f3a..."       // 唯一水印，泄露溯源
}

版本号有两层：schemaVersion（License 结构演进）与 licenseVersion（同一 License 每次变更 +1，用于历史追溯）；同时 appVersionRange 控制被授权软件的版本范围，满足"版本授权"需求。

3. 硬件指纹（机器绑定）

machineCode = SM3/SHA256( 归一化(CPU序列号 + 主板UUID + 网卡MAC + 磁盘序列号 + 机器主机名) )

维度风险/对策
选取因子多因子组合，避免单因子（MAC 可改）
虚拟化/容器云上无稳定硬件 ID → 提供"实例 ID / K8s 节点标识 / 容器 UUID + 厂商颁发的激活码"作为替代指纹
硬件变更LOOSE 策略：N 个因子允许 M 个变化（如 5 取 4 匹配即放行），换网卡不致失效
指纹算法下沉 native（02 文档），防止伪造指纹

4. 离线授权流程

客户机                          运营控制台/后端                KMS/HSM
  │  ① 运行指纹采集工具            │                            │
  │  生成 machineCode             │                            │
  │ ───── machineCode + 订单 ───▶ │                            │
  │                               │ ② 选租户/套餐/版本/有效期    │
  │                               │ 组装 claims                 │
  │                               │ ───── 待签 claims ───────▶ │ ③ 私钥签名
  │                               │ ◀──────── 签名 ──────────── │
  │                               │ ④ 落库(版本+1, 审计)         │
  │ ◀──────── .lic 文件 ───────── │                            │
  │ ⑤ 导入 .lic                   │                            │
  │ ⑥ SDK 验签+校验指纹/期限/版本   │                            │
  │ ⑦ 通过→放行 / 失败→拒绝         │                            │
  • 校验项：签名有效 ∧ machineCode 匹配 ∧ 当前时间 ∈ [notBefore,notAfter] ∧ 软件版本 ∈ appVersionRange ∧ 功能门禁。
  • 优点：无需联网，适合内网/涉密环境。
  • 缺点：无法实时吊销、无法精确计量 → 用有效期 + 到期续签 + 可选的离线用量回传弥补。

⚠️ 关于硬件指纹绑定的现实性：上述"采集指纹→回传→人工签发→回传→导入"的硬绑定流程运维成本高、对云/容器不友好、客户体验差，对中小团队（小厂）并不现实。 machineCode 绑定在本方案中应视为面向高价值/涉密离线场景的可选增强项，而非默认离线方案。 面向小厂的、低成本可落地的离线/弱在线授权替代方案，见 07-小厂离线授权现实方案.md。

5. 在线授权流程

应用启动 → 激活(activation) → 周期心跳(heartbeat) → 续期/吊销实时生效

激活

  1. 应用首次启动，携带激活码/初始 License + machineCode 调用 POST /activate。
  2. 服务端校验额度（并发席位、是否已激活满）→ 绑定实例 → 返回服务端签名的授权快照。

心跳（防重放 + 防共享）

SDK → 服务端：{ instanceId, machineCode, nonce, ts, hmac }
服务端校验：nonce 未用过(防重放) + 席位未超 + 未吊销
服务端 → SDK：{ authSnapshot, serverTime, signature }   // 服务端签名
SDK：验签 + 更新本地授权 + 同步可信时间
  • 并发/浮动授权：服务端用 Redis 维护活跃 instanceId 集合 + TTL；超过 concurrency 拒绝新实例，实现"同时在线 N 个"的浮动席位。
  • 实时吊销：服务端维护 CRL（吊销列表），心跳即时下发；被吊销 → SDK 立即降级/停服。
  • 通道安全：mTLS 双向证书 + 请求签名（HMAC/nonce）防中间人与重放。

断网容错（关键）

  • 心跳失败进入宽限期 gracePeriodHours：期间凭最近一次合法快照继续运行（fail-open 有限期）。
  • 宽限期耗尽仍无法联通 → 核心功能降级或停服（fail-closed），避免误伤生产同时防止永久离线绕过。
模式优点缺点适用
离线不依赖网络、适合涉密内网不能实时吊销/计量内网、政企、私有化
在线实时吊销、浮动席位、精确计量、远程停用依赖网络SaaS、有外网环境
混合(HYBRID)离线兜底 + 在线增强实现复杂推荐默认：平时在线，断网走宽限期/离线 License

攻击防护

7. License 生命周期、版本号与历史追溯

7.1 状态机

DRAFT(草稿) → ISSUED(已签发) → ACTIVATED(已激活)
   → RENEWED(已续期) ↺            → SUSPENDED(暂停) ↔ ACTIVATED
   → MODIFIED(已变更, 版本+1) ↺
   → REVOKED(已吊销, 终态)
   → EXPIRED(已过期, 终态)

7.2 版本与历史（核心需求）

  • 每次签发/续期/变更/吊销都使 licenseVersion + 1，并写入 license_history 表，保存完整快照（claims JSON + 签名 + 操作人 + 时间 + 变更原因 + 关联订单）。
  • 控制台支持：
    • 版本列表与任意两版 diff（高亮新增/失效的模块、功能、有效期、席位变化）。
    • 当前生效版本标识；历史版本可查看不可改（不可变审计）。
    • 吊销记录与原因，进入 CRL。
  • 审计日志（audit_log）：记录每一次签发、下载、激活、心跳异常、吊销，含操作人/IP/时间，支撑合规追溯与法务取证。

7.3 运营能力

  • 到期前 30/15/7 天自动预警（站内+邮件+短信）。
  • 批量签发（按订单/客户清单）、续期一键延期、变更（加购模块/扩席位即时重签）。
  • License 与订单/合同关联，形成"销售→授权→交付→续费"闭环。

8. 客户端 SDK 设计

license-sdk (jar)
├── fingerprint      硬件指纹采集（JNI）
├── verifier         公钥验签 + claims 校验（关键逻辑 native）
├── store            离线 .lic 加载 / 在线快照缓存（加密）
├── online           激活 / 心跳 / CRL / 可信时间（mTLS）
├── clock            时间回拨检测（highWaterMark）
├── gate             功能/版本门禁 API + 注解
└── spi              事件回调（即将到期 / 已降级 / 已吊销）

功能门禁用法（开发者友好）

// 注解式：方法级功能授权
@RequiresFeature("REPORT.EXPORT")
public byte[] exportReport(...) { ... }

// 编程式：运行期判断
if (License.has("RISK.REALTIME") && License.appVersionAllowed()) { ... }

// 配额：
License.checkQuota("MAX_USERS", currentUserCount);
  • 通过 Spring AOP 拦截 @RequiresFeature，未授权抛 LicenseDeniedException 并返回标准错误码。
  • 启动时校验一次（fail-closed），运行期周期性复检（在线心跳/离线时间校验）。
  • 提供 LicenseStatus 端点（受限暴露）给运维查看授权状态、到期时间、已用席位。

9. 潜在风险与应对

风险应对
客户端被逆向后伪造 License非对称签名（私钥不出域）从根上杜绝伪造
patch 掉验签直接放行校验 Native 化 + 完整性自校验 + 解密密钥依赖 License（02 文档 §4）
在线服务宕机/断网影响生产宽限期 fail-open + 离线 License 兜底；License Server 高可用多活

私钥泄露导致全部 License 可伪造KMS/HSM 托管 + 密钥分级（按产品/大版本）+ 可轮换/吊销 + 重签机制
