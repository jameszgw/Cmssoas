02 · 代码保护方案（防逆向 / 防泄露）

目标：对客户交付的 Spring Boot jar 进行纵深防护，使反编译、脱壳、破解 License 校验的成本远高于软件本身价值。

1. 设计思路：纵深防御分层

单一手段都能被针对性突破，因此采用"洋葱式"多层防护，攻击者必须逐层攻克：

┌────────────────────────────────────────────────────────┐
│ L5 法律与运营层：LICENSE 协议、合同、在线吊销、水印追溯       │
│ ┌────────────────────────────────────────────────────┐ │
│ │ L4 运行环境层：反调试、反 dump、完整性自校验、防 attach  │ │
│ │ ┌────────────────────────────────────────────────┐ │ │
│ │ │ L3 关键逻辑层：License 校验/核心算法 Native 化      │ │ │
│ │ │ ┌────────────────────────────────────────────┐ │ │ │
│ │ │ │ L2 字节码加密层：Xjar/ClassFinal 套壳 + 运行解密 │ │ │ │
│ │ │ │ ┌────────────────────────────────────────┐ │ │ │ │
│ │ │ │ │ L1 混淆层：名称/控制流/字符串/反射代理     │ │ │ │ │
│ │ │ │ │ ┌────────────────────────────────────┐ │ │ │ │ │
│ │ │ │ │ │ L0 业务源码                          │ │ │ │ │ │
│ │ │ │ │ └────────────────────────────────────┘ │ │ │ │ │
│ │ │ │ └────────────────────────────────────────┘ │ │ │ │
│ │ │ └────────────────────────────────────────────┘ │ │ │
│ │ └────────────────────────────────────────────────┘ │ │
│ └────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────┘
层防护手段主要对抗
L1 混淆类/方法/字段重命名、控制流平坦化、字符串加密、无用代码注入反编译后可读性、静态分析
L2 字节码加密整包/类文件加密，运行期内存解密直接 jar -> class 反编译、静态提取
L3 Native 化License 校验、密钥派生、核心算法用 JNI/GraalVM 编译为原生码patch 校验逻辑、Java 层 hook
L4 运行环境反调试（JDWP 检测）、反 attach、内存 dump 抑制、jar 完整性 hash 校验动态调试、内存脱壳、agent 注入
L5 法律/运营授权协议、唯一水印、在线吊销、审计追溯即便破解也可发现与追责

2. 技术选型对比

2.1 代码混淆器

工具授权能力对 Spring 兼容性评价
ProGuard免费/开源重命名、瘦身、基础优化；无控制流/字符串加密反射/CGLIB 需大量 keep 规则，易踩坑入门免费方案，防护强度有限
yGuard免费/开源重命名为主一般简单项目可用
Allatori商用重命名 + 控制流混淆 + 字符串加密 + 水印 + 资源加密较好，有 Spring 实践推荐，性价比高，防护强
DashO商用（较贵）强混淆 + 防篡改 + 反调试好预算充足可选

结论：免费起步用 ProGuard（注意 keep Spring 反射/实体/MyBatis Mapper/Controller）；商用首选 Allatori（字符串加密对隐藏 License 校验、密钥、URL 很关键）。

⚠️ Spring Boot 混淆注意：Bean 名、@Configuration、@Entity 字段（影响 ORM 映射）、Controller 方法（影响路由）、Jackson 序列化字段、SPI/META-INF 等需配置 keep 白名单，否则运行时反射失败。建议只混淆核心业务/算法包，放过框架交互边界。

2.2 字节码加密 / 套壳

工具授权原理优点缺点/风险
Xjar开源(国产)对 jar 内 class/资源 AES 加密，生成 Go 编写的解密启动器（go-xjar），运行时解密加载整包保护、启动器为原生不易逆、可绑定密码启动器与密钥分发需保护；解密后类仍可能被内存 dump
ClassFinal开源(国产)class 加密，运行期用自定义 ClassLoader + javaagent 内存解密；支持机器绑定 + 密码 + 防 agent集成简单、Maven 插件、支持机器绑定维护活跃度一般；内存 dump 风险同上
自研 ClassLoader + Agent自有自定义加密格式 + native agent 解密 + 反 dump可控、可与 License 密钥联动开发成本高

结论：推荐 Xjar（整包套壳） 或 ClassFinal（类级 + 机器绑定）。两者均可，关键在于解密密钥不能硬编码在 jar 里——应由 License 派生 或运行时从受保护渠道获取（见 §4）。

2.3 关键逻辑 Native 化

方案原理防护强度成本
GraalVM native-image将（部分或整体）应用 AOT 编译为本地可执行文件，无 .class 字节码高（无字节码可反编译）中-高：反射/动态代理需配置 reflect-config.json，Spring Boot 有官方 AOT 支持但需适配
JNI (C/C++)把 License 验签、密钥派生、指纹算法用 C 实现，编译为 .so/.dll高（针对最关键逻辑）中：需跨平台编译、混淆 native（OLLVM）
jar 转 exe（jpackage/launch4j/exe4j）仅打包封装，不改变字节码低（不是防护，只是分发便利）低

结论：把最敏感的逻辑（License 验签、解密密钥派生、硬件指纹）下沉到 JNI native 库，并用 OLLVM 混淆 native 代码。整体应用可选 GraalVM native-image 进一步消除字节码。 明确澄清：jar 转 exe 本身不提供防逆向能力，不能作为主力防护，仅用于分发体验。

2.4 选型小结（推荐组合）

业务源码
  └─(L1) ProGuard/Allatori 混淆（核心包，字符串加密）
        └─(L3) 关键校验/密钥逻辑 → JNI native(.so/.dll, OLLVM 混淆)
              └─(L2) Xjar/ClassFinal 加密套壳（密钥由 License 派生）
                    └─(L4) native agent：反调试 + 反 dump + 完整性校验
                          └─(可选) GraalVM native-image 整体原生化

3. 在 CI/CD 中的构建流水线

mvn package（原始 boot jar）
   ↓
① 混淆（ProGuard/Allatori Maven 插件，keep 白名单）
   ↓
② 编译/链接 native 库（JNI 关键逻辑，OLLVM）→ 内置到 jar
   ↓
③ 加密套壳（Xjar/ClassFinal 插件）→ 产出受保护交付物
   ↓
④ 完整性签名：对最终产物计算 hash 并由厂商私钥签名，启动时自校验
   ↓
⑤ 注入唯一水印（客户/批次标识）用于泄露溯源
   ↓
产出：客户专属、可溯源的受保护交付包
  • 流水线在隔离的构建机执行，混淆映射表（mapping）、native 源码、密钥不随交付物分发，仅留存于厂商侧（mapping 用于崩溃栈反混淆排障）。
  • 每个客户/批次产出唯一水印，一旦在外网发现泄露包，可溯源到泄露方。

4. 与 License 的联动（核心创新点）

让"解密密钥"依赖"合法 License"，把代码保护与授权强绑定：

  1. 套壳加密时，class 解密密钥 = KDF(产品根密钥, License 中的 licenseId/指纹)。
  2. 应用启动 → native agent 先校验 License（验签 + 指纹 + 有效期）→ 校验通过才派生出正确的解密密钥 → 才能解密加载业务类。
  3. 效果：没有合法 License，连业务字节码都无法被正确解密运行；单纯 patch 掉 Java 层校验也无法得到正确密钥。

这一设计把"破解 License"与"脱壳解密"两个难题耦合在一起，显著提升破解门槛。

5. 反调试 / 反 dump / 完整性校验（L4）

手段实现对抗
反调试检测 JDWP（-agentlib:jdwp、-Xdebug）、检测 java.lang.management 中的调试参数、native 层 ptrace 自附加动态断点调试
反 attach启动后调用 native 关闭/监控 attach 机制，检测异常 -javaagent/instrument运行期注入 agent dump 内存类
内存 dump 抑制native agent 监控可疑内存读取、关键类用后即清/不长期驻留明文从内存提取已解密 class
完整性自校验启动与运行期周期性校验 jar/native 库 hash 是否被篡改（与厂商签名比对）patch 字节码/替换 native 库
多点冗余校验License 校验结果在多处分散验证、结果参与业务计算单点 NOP 绕过

失败策略：检测到调试/篡改 → 不直接崩溃（易被定位），而是延迟、随机化失败或静默降级，增加攻击者定位难度（tamper-evident 优于 tamper-fail）。

6. 潜在风险与应对

风险说明应对措施
字节码终可反编译Java 本质可逆，混淆只增加阅读难度叠加加密 + Native + 在线校验；核心价值逻辑 Native 化；接受"抬高门槛"的现实目标
内存 dump 脱壳自定义 ClassLoader 解密后的 class 在内存中可被 dump（如 HSDB、自写 agent）native 反 dump agent；关键类不长期明文驻留；最核心逻辑放 Native（dump 出来也是机器码）；GraalVM 整体原生化
patch 校验逻辑攻击者 NOP 掉验签返回 true校验 Native 化 + 完整性自校验 + 多点冗余 + 解密密钥依赖 License（§4）
Spring 反射兼容性混淆破坏 Bean/ORM/路由精确 keep 白名单；只混淆核心包；CI 中加冒烟测试验证混淆包可启动
GraalVM 适配成本反射/动态代理/AOT 配置复杂仅对核心模块原生化；用 Spring Boot AOT + 自动生成 reflect-config；预留适配工时
native 库被逆向.so/.dll 仍可被 IDA/Ghidra 分析OLLVM 控制流混淆 + 字符串加密 + 反调试；关键常量运行期派生
性能开销加密解密、完整性校验、心跳带来开销类解密只在加载时一次；完整性校验异步/抽样；native 校验高性能
mapping/密钥泄露反混淆映射表或根密钥泄露则防护失效构建机隔离、密钥入 KMS/HSM、最小权限、定期轮换、不随交付物分发
跨平台分发native 库需多平台编译CI 多平台交叉编译（Linux/Win/麒麟/统信），按客户环境裁剪
误杀正常运维反调试误判客户 APM/监控 agent提供白名单配置、官方支持的可观测接入方式、宽松/严格模式可配

7. 交付与排障

  • 交付物：受保护 jar/native 包 + 启动器 + License 导入工具 + 指纹采集工具。
  • 排障：保留 ProGuard mapping，崩溃栈用 retrace 反混淆；native 崩溃保留符号表（厂商侧）。
  • 降级开关：提供"调试构建"（关闭部分防护）供联调，严禁外发，CI 区分 debug / release profile。

附:在线代码加固(运营平台,与构建/打包并存)

除"构建/打包时加固"(本文上述 ProGuard/加密方案,见 examples/protected-app)外,运营平台还提供 在线加固:用户上传 jar → 平台异步加固 → 下载。两者并存、不取代,由运营用户按租户设置模式 (BUILD/ONLINE/BOTH)并可在任务级覆盖。技术含 ProGuard 混淆、关键类加密+License 绑定、口令式类加密, 经 HardenProvider 抽象可插拔接商用器。详见 docs/功能-在线代码加固.md。
