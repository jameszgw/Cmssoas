# 实施方案：在线用户须知 / 用户授权 / 合同签约 / 智能客服

> 面向下一步工作的设计方案。复用现有架构:Spring Boot 后端 + Vue3 控制台 + 多租户(Schema 隔离) + RBAC + Flyway + Nacos/CentOS 部署 + 发件箱(outbox) + 审计。每项均含:目标场景 / 数据模型 / 流程 / 接口 / 前端 / 选型 / 集成点 / 合规风险 / 工作量。

## 总览与集成原则
- **统一鉴权**:新接口纳入 JWT + `@RequirePerm`;面向租户管理员/最终用户的公开页(须知阅读、签署、客服)走专用放行(类比已有 `/api/activation/**`)。
- **统一审计**:所有“同意 / 签署 / 撤回 / 客服会话”写入 `audit_log`(`AuditWriter`)。
- **统一通知**:邮件复用 `outbox`(异步+重试);可叠加企业微信 webhook。
- **统一前端**:复用换肤/i18n/PageHelp/操作帮助/blob 下载。
- **多租户**:协议版本、合同、客服会话均带 `tenant_code`,隔离查询。

## 一、在线用户须知(服务条款 / 公告 / 用户协议展示)
**目标**:集中管理“用户须知 / 服务条款 / 隐私政策 / 公告”,支持富文本、版本化;在首次登录或版本更新时强制阅读确认。与“用户授权”配对(须知=内容,授权=同意记录)。

| 项 | 设计 |
|---|---|
| 数据模型 | `notice(id, type[TERMS/PRIVACY/NOTICE/ANNOUNCEMENT], title, content_html, version, status[DRAFT/PUBLISHED/ARCHIVED], effective_at, force_ack, created_at)` |
| 后端接口 | `GET /api/notices`(管理列表)、`POST/PUT /api/notices`(编辑,`notice:edit`)、`POST /api/notices/{id}/publish`;公开:`GET /pub/notices/active?type=`(取当前生效版本) |
| 前端 | 管理后台“须知管理”页(富文本编辑器,如 wangEditor/Quill);用户侧弹层/独立页展示,`force_ack=true` 时阻断使用直至确认 |
| 选型 | 富文本:Quill(轻量);版本:每次发布生成新 version,旧版归档保留可追溯 |
| 集成点 | 登录后/激活后校验“是否已确认当前生效版本”,未确认→弹须知(复用 ForcePwdDialog 的阻断模式) |
| 合规 | 内容版本不可变;留存历史版本与生效时间,满足“以用户当时同意的版本为准” |
| 工作量 | **S(1–2 天)**,与“用户授权”合并开发更高效 |

## 二、用户授权(同意管理 Consent)
**目标**:记录“谁、何时、对哪个协议版本、以何方式”表示同意/撤回(服务条款、隐私政策、数据处理、营销推送等),可查询、可撤回、可导出,满足合规(个保法/GDPR 思路)。

> 备注:若“用户授权”指**第三方 OAuth 授权登录**(微信/企业微信扫码登录),见文末“变体”。本节按**协议同意/数据授权**实现。

| 项 | 设计 |
|---|---|
| 数据模型 | `user_consent(id, tenant_code, subject[用户标识], notice_id, notice_type, version, action[GRANTED/REVOKED], channel[WEB/EMAIL/API], ip, user_agent, created_at)` |
| 后端接口 | `POST /pub/consents`(提交同意:notice_id+version)、`POST /pub/consents/revoke`、`GET /api/consents`(管理查询,`consent:view`)、`GET /api/consents/export.csv`(复用 CsvUtil) |
| 前端 | 须知弹层底部“我已阅读并同意”勾选→提交同意;个人中心“我的授权”可查看/撤回 |
| 集成点 | 与“须知 force_ack”联动:确认即写 consent;激活流程纳入隐私同意;营销同意作为发信前置校验 |
| 合规 | 记录 IP/UA/版本/时间(已有审计风格);撤回后停止对应数据处理/营销;导出供取证 |
| 风险 | 主体标识:租户管理员用 `ops_user`,最终用户需明确主体口径(邮箱/手机号) |
| 工作量 | **S–M(2–3 天)** |

## 三、合同签约(电子签)
**目标**:基于订阅/套餐自动生成合同,发起多方电子签署,签署完成后存证归档;与订阅/账单/License 闭环。

**选型对比**:
| 方案 | 说明 | 适用 |
|---|---|---|
| **第三方合规电子签**(e签宝 / 法大大 / DocuSign)| API 对接,具备 CA 实名 + 数字证书 + 可信时间戳 + 司法存证,**法律效力最强** | **推荐**(对外正式合同) |
| 自建签章 | PDF 模板套打 + 自有证书签名(BC,可复用 SM2)+ RFC3161 时间戳 + 哈希存证 | 内部留痕/低成本,法律效力弱于第三方 |

| 项 | 设计 |
|---|---|
| 数据模型 | `contract_template(id, name, content_html/pdf, variables)`;`contract(id, no, tenant_code, customer, subscription_id, template_id, amount, status[DRAFT/SENT/SIGNING/SIGNED/VOID], provider, provider_flow_id, file_id, created_at, signed_at)`;`contract_party(contract_id, name, role, email/phone, sign_status, signed_at)` |
| 流程 | 订阅创建→按模板生成合同(填充租户/套餐/金额/期限)→发起签署(第三方:创建签署流程,返回签署链接;短信/扫码)→各方签字→回调更新状态→存证(哈希+时间戳)→归档(File/对象存储)→联动账单/License |
| 后端接口 | `POST /api/contracts`(生成,`contract:edit`)、`POST /api/contracts/{id}/send`(发起签署)、`POST /api/contracts/callback`(第三方回调,验签放行)、`GET /api/contracts`、下载 PDF(blob) |
| 前端 | “合同管理”页:模板、合同列表、状态流转、发起签署、下载;签署方走第三方页面或内嵌签署组件 |
| 集成点 | 订阅→合同→签署完成→自动出账/激活 License;退订→合同作废 |
| 合规 | 实名认证、CA 证书、可信时间戳、防篡改哈希、签署轨迹存证;原文不可改 |
| 风险 | 第三方依赖网络出网(CentOS 网络策略需放行其域名 + 回调地址公网可达);需账号与计费 |
| 工作量 | 自建 **M(3–5 天)**;第三方 **M–L(5–8 天,含联调与回调)** |

## 四、智能客服(基于 Claude)
**目标**:控制台/官网内嵌 AI 客服,回答产品/授权/订阅/账单问题;能**查询真实数据**(License/订单/在线状态),命中知识库(文档/FAQ)作答;支持流式、转人工、审计与脱敏。

**技术选型(已核对 Claude API)**:
- **SDK**:Java 官方 `com.anthropic:anthropic-java`(后端调用,密钥不出域);客户端浏览器**不直连** Anthropic。
- **模型**:默认 `claude-opus-4-8`(最强);高并发/低成本场景可选 `claude-sonnet-4-6`(成本权衡,需显式选择)。
- **思考**:`thinking: {type: "adaptive"}`(自适应)。
- **工具调用(Tool Use / BetaToolRunner)**:把现有服务暴露为工具,让模型按需查实时数据:
  - `get_license_status(licenseId)` → 复用 LicenseService
  - `get_subscription(tenantCode)` / `get_invoices(tenantCode)` → 复用 Subscription/Invoice
  - `get_online_instances(licenseId)` → 复用 OnlineService
  工具在后端执行,模型只拿结果,**不接触数据库与密钥**。
- **知识库 RAG**:`docs/` + FAQ 切片为知识库;按问题召回相关片段注入提示;对**稳定的系统提示+知识前缀启用 prompt caching**(`cache_control: ephemeral`),显著降本降延迟。初版可用关键词/BM25 召回,后续可接向量检索。
- **流式**:后端 SSE 把增量 token 推给前端聊天窗(复用前端流式渲染)。
- **会话**:多轮历史由后端维护(API 无状态,每次回传历史)。

| 项 | 设计 |
|---|---|
| 数据模型 | `cs_conversation(id, tenant_code, user_ref, status[OPEN/ESCALATED/CLOSED], created_at)`;`cs_message(id, conversation_id, role[user/assistant/tool], content, tokens, created_at)` |
| 后端接口 | `POST /api/cs/chat`(SSE 流式;入参:会话id+用户消息)、`POST /api/cs/escalate`(转人工)、`GET /api/cs/conversations`(管理/审计) |
| 前端 | 右下角悬浮客服窗(气泡/输入框/流式渲染/“转人工”);可复用现有 UI 风格与 i18n |
| 安全/脱敏 | **不向 API 外发**:私钥、密码、完整密钥串、敏感 PII;工具结果做字段白名单;`ANTHROPIC_API_KEY` 经 Nacos/KMS 注入;出网经 CentOS 网络策略放行 `api.anthropic.com`(或走代理) |
| 护栏 | 系统提示限定“仅回答本产品相关”;拒绝越权操作(只读工具,不做签发/吊销等写操作,或写操作必须人工确认);限流(复用 SdkRateLimit 思路) |
| 审计 | 会话与消息留存;敏感操作不可由 AI 自动执行 |
| 工作量 | **M–L(4–7 天)**:工具封装 + RAG 召回 + SSE + 前端窗 + 护栏 |

**核心调用示意(Java,后端)**:
```java
// 默认 claude-opus-4-8 + 自适应思考 + 工具(查实时数据)+ 知识库前缀(prompt caching)
// 工具用 BetaToolRunner 注册:get_license_status / get_subscription / get_invoices / get_online_instances
// 系统提示 + 知识库片段放前缀并 cache_control: ephemeral；用户问题放最后
// 输出走 streaming，经后端 SSE 转发到前端聊天窗
```

## 实施优先级与工作量
| 优先级 | 项 | 价值 | 工作量 | 依赖 |
|---|---|---|---|---|
| P0 | 须知 + 用户授权(同意) | 合规刚需,改造小 | S–M | 无(合并做) |
| P1 | 智能客服 | 体验/降本,亮点 | M–L | ANTHROPIC_API_KEY + 出网 |
| P1 | 合同签约 | 商业闭环 | M(自建)/L(第三方) | 第三方账号/出网/回调公网 |

建议顺序:**① 须知+授权 → ② 智能客服 → ③ 合同签约**(签约第三方联调最重,放最后)。

## 共性:安全 / 合规 / 部署
- 密钥(Anthropic、电子签、短信)统一入 Nacos/KMS,不落明文(见 `docs/密钥管理-KMS.md`)。
- 出网:CentOS 网络策略放行第三方域名与回调;回调地址需公网可达 + 验签。
- 隐私:同意/合同/会话留存最小化、可导出、可删除(配合撤回)。
- 复用:审计、outbox 邮件、CSV 导出、blob 下载、RBAC、i18n、换肤。

## 变体:用户授权 = 第三方 OAuth 登录(可选)
若“用户授权”指扫码登录(微信/企业微信),则:`POST /pub/oauth/{provider}/callback` 换取用户信息 → 绑定/创建账号 → 签发 JWT;与现有登录并存。与上文“同意管理”不冲突,可叠加。
