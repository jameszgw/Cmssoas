00 · 完整解决方案（分步骤 · 含前后端）

本文是权威主文档：按实施步骤推进，每一步统一包含： ① 设计思路 ② 技术选型对比 ③ 潜在风险 ④ 应对措施，并标注前端 / 后端落点。 细节展开见附录 01–07。

目标与三大子系统

  1. 代码保护（防逆向、防泄露）——步骤 1–5
  2. License 认证（离线 + 在线、可管理、版本与历史追溯）——步骤 6–12
  3. 多租户运营平台（前后端、租户开通、DB 初始化、超管、模块/功能/版本授权）——步骤 13–19
  4. 收尾：密钥、部署、路线图——步骤 18–20

⚠️ 现实边界：纯软件无法绝对防破解；目标是抬高破解成本 > 软件价值，以"技术 + 在线吊销/水印溯源 + 法律协议"构成综合防线。

第一部分 · 代码保护

步骤 1：代码混淆

① 设计思路 仅混淆核心业务/算法包（风控规则、计费、License 校验辅助逻辑），放过框架交互边界（Controller 路由、@Entity 字段、Bean 名、Jackson 序列化字段、SPI），避免破坏 Spring 反射。开启名称重命名 + 控制流平坦化 + 字符串加密，重点把"License 校验、密钥、URL 常量"等敏感字符串加密。

② 技术选型对比

工具授权能力Spring 兼容结论
ProGuard免费重命名/瘦身，无控制流/字符串加密需大量 keep起步可用
yGuard免费重命名为主一般简单项目
Allatori商用重命名+控制流+字符串加密+水印较好推荐
DashO商用贵强混淆+防篡改好预算足可选

③ 潜在风险 / ④ 应对措施

风险应对
混淆破坏反射致启动失败精确 keep 白名单；只混淆核心包；CI 加"混淆后冒烟启动测试"
防护强度有限（仍可反编译）叠加后续步骤（加密+Native）；字符串加密隐藏关键逻辑
崩溃栈不可读、排障难保留 mapping（仅厂商侧），用 retrace 反混淆

前后端落点：后端（被保护的 Spring Boot 交付物构建期）；前端无关。

步骤 2：字节码加密 / 套壳

① 设计思路 对 jar 内 class/资源加密，运行期内存解密加载。关键创新：class 解密密钥 = KDF(产品根密钥, License 信息) —— 没有合法 License 就解不出正确密钥、跑不起业务字节码（与步骤 6–9 联动）。

② 技术选型对比

工具授权原理优点风险
Xjar开源(国产)整包 AES 加密 + Go 原生启动器解密整包保护、启动器难逆解密后类可被内存 dump
ClassFinal开源(国产)class 加密 + 自定义 ClassLoader/agent 内存解密，支持机器绑定集成简单、Maven 插件维护活跃度一般
自研 ClassLoader+Agent自有自定义格式 + native 解密可控、可与 License 联动开发成本高

③ 潜在风险 / ④ 应对措施

风险应对
解密后内存 dump 脱壳步骤 3 把核心逻辑 Native 化；步骤 4 反 dump；明文类不长期驻留
解密密钥硬编码被提取密钥由 License 派生（不入包）；根密钥入 KMS、构建机隔离
启动器/agent 被替换步骤 4 完整性自校验（hash + 厂商签名比对）

前后端落点：后端（构建期 + 运行期）；前端无关。

步骤 3：关键逻辑 Native 化

① 设计思路 把License 验签、解密密钥派生、硬件/软指纹算法、核心算法用 JNI（C/C++）实现并编译为 .so/.dll，再用 OLLVM 混淆 native；可选用 GraalVM native-image 对核心模块整体 AOT，彻底消除可反编译字节码。jar 转 exe 仅作分发便利、不计入防护。

② 技术选型对比

方案原理防护成本
JNI(C/C++)+OLLVM关键逻辑原生化高（针对最敏感点）中：跨平台编译
GraalVM native-image整体/模块 AOT，无字节码高中-高：反射需 reflect-config
jpackage/launch4j/exe4j仅封装，不改字节码无（非防护）低

③ 潜在风险 / ④ 应对措施

风险应对
native 仍可被 IDA/Ghidra 逆向OLLVM 控制流混淆 + 字符串加密 + 关键常量运行期派生
跨平台编译维护重CI 多平台交叉编译（Linux/Win/麒麟/统信），按客户裁剪
GraalVM 适配反射/代理复杂仅核心模块原生化；用 Spring Boot AOT 自动生成配置；预留工时

前后端落点：后端（被保护应用 + SDK 的 native 部分）；前端无关。

步骤 4：运行时防护（反调试 / 反 dump / 完整性自校验）

① 设计思路 native agent 在运行期检测 JDWP/调试参数、ptrace 自附加、异常 -javaagent 注入；周期性校验 jar/native 库 hash 与厂商签名是否一致；License 校验结果多点冗余并参与业务计算。检测到篡改不立即崩溃，而是延迟/随机化失败/静默降级（tamper-evident 优于 tamper-fail）。

② 技术选型对比

手段实现对抗
反调试检测 JDWP/-Xdebug + native ptrace动态断点
反 attach关闭/监控 attach、检测可疑 agent运行期注入 dump
完整性校验hash + 厂商私钥签名比对patch 字节码/换 native
多点冗余校验校验结果分散、参与计算单点 NOP 绕过

③ 潜在风险 / ④ 应对措施

风险应对
误杀客户 APM/监控 agent提供白名单 + 官方可观测接入；严格/宽松模式可配
检测点被定位逐一拔除多点冗余 + 延迟随机失败 + 结果耦合业务
校验自身被 patch校验逻辑 Native 化（步骤 3）+ 互相校验

前后端落点：后端（被保护应用运行期）；前端无关。

步骤 5：CI/CD 安全构建流水线

① 设计思路 在隔离构建机串联：mvn package → 混淆 → 编译 native → 加密套壳 → 完整性签名 → 注入客户唯一水印，产出客户专属、可溯源交付物。mapping、native 源码、根密钥不随交付物分发，仅留厂商侧。区分 debug（关闭部分防护，仅内部联调）/release profile。

② 技术选型对比

环节选型备选
构建Maven + GitLab CI/JenkinsGradle
混淆插件ProGuard/Allatori Maven 插件——
加密插件Xjar/ClassFinal Maven 插件自研
签名KMS 签名（私钥不出域）本地 keystore（不推荐）

③ 潜在风险 / ④ 应对措施

风险应对
构建机泄露 mapping/根密钥构建机隔离、最小权限、密钥入 KMS、定期轮换
debug 包外泄profile 隔离 + 制品库权限 + 水印区分
泄露包难溯源每客户/批次唯一水印 → 外网发现可定位泄露方

前后端落点：后端（交付物构建）；可在运营平台（前端）提供"按客户出包/水印管理"界面。

第二部分 · License 认证

步骤 6：信任根与签名体系

① 设计思路 非对称签名为信任根：厂商私钥签发（KMS/HSM 不出域），客户端内置公钥验签。客户端永不持私钥 → 无法伪造 License，只能尝试 patch（由第一部分对抗）。客户端内置多公钥 + kid，支持密钥轮换。

② 技术选型对比

算法签名长度性能国密结论
RSA-2048长验快签慢否兼容性好
ECDSA P-256短快否可选
Ed25519短最快否通用推荐
SM2短快是信创推荐

③ 潜在风险 / ④ 应对措施

风险应对
私钥泄露→全线可伪造KMS/HSM 托管；按产品/大版本分级密钥（缩小爆炸半径）；可轮换/重签
验签被 patchNative 化 + 完整性校验 + 解密密钥依赖 License

前后端落点：后端（签发引擎调 KMS）；前端无（公钥在 SDK）。

步骤 7：License 数据模型与"模块/功能/版本"授权

① 设计思路 License claims 含：licenseId / schemaVersion / licenseVersion / tenantId / 客户 / 模块 / 功能+配额 / appVersionRange（软件版本授权）/ 有效期 / mode / watermark / 绑定信息。两层版本号：schemaVersion（结构演进）、licenseVersion（每次变更+1，供历史追溯）；appVersionRange 实现按版本授权。

② 技术选型对比

载体选型备选
离线签名 .lic（claims+签名，类 JWS 自定义）TrueLicense（密钥随包易提取，不推荐高价值）
在线API 返回服务端签名快照——
功能门禁注解 @RequiresFeature + Spring AOP编程式判断

③ 潜在风险 / ④ 应对措施

风险应对
篡改 claims签名覆盖全字段，改一字节即失败
schema 升级不兼容旧 LicenseschemaVersion 兼容、SDK 向后兼容
授权与平台配置不一致套餐→License 单一映射来源，变更即重签

前后端落点：后端（模型/签发）；前端（控制台配置模块/功能/版本矩阵——见步骤 15、17）。

步骤 8：离线授权（含小厂现实方案）

① 设计思路 默认采用"签名期限 License（不绑机）" —— 只签客户+功能+版本+有效期+水印，防伪造靠签名、防滥用靠水印溯源+到期续签+合同。硬件指纹绑定仅作高价值/涉密场景可选增强（详见附录 07，含小厂 T0–T3 分级方案）。

② 技术选型对比（小厂离线分级）

档方案绑机联网运维防复制
T0 ⭐签名期限(不绑机)+水印否否极低弱
T1软节点锁(installationId)软否低中
T2 ⭐在线自助激活+离线运行软激活时一次低(自动)强
T3 ⭐Keygen 自托管/SaaS可配可配低(不自研)强
(增强)硬件指纹绑定硬否高强

③ 潜在风险 / ④ 应对措施

风险应对
硬件指纹运维重、云容器漂移、客户抵触不默认硬绑；用 T0/T2/T3
不绑机可复制多部署唯一水印溯源 + 有效期强制续 + 合同 + 可选遥测发现异常副本数
无法离线实时吊销到期不续 + 法律；需实时吊销则上 T2/在线

前后端落点：后端（签发/激活服务）；前端（控制台一键/批量签发、激活码管理、离线签发导出）。

步骤 9：在线授权（激活 / 心跳 / 吊销 / 浮动席位）

① 设计思路 应用激活后周期心跳；服务端校验有效性、并发席位（Redis 活跃实例 TTL）、CRL 实时吊销，返回服务端签名快照。通道 mTLS + nonce 防重放。断网进入宽限期 fail-open，超期 fail-closed，避免误伤又防永久离线绕过。推荐 HYBRID：平时在线、断网走宽限期/离线兜底。

② 技术选型对比

模式优点缺点适用
离线不依赖网络不能实时吊销/计量内网/涉密
在线实时吊销/席位/计量/远程停用依赖网络SaaS/有外网
HYBRID ⭐离线兜底+在线增强实现复杂默认推荐

③ 潜在风险 / ④ 应对措施

风险应对
断网致生产停摆宽限期 + 离线兜底 + 服务高可用多活
心跳重放/抓包伪造响应nonce+时间戳+去重；mTLS；服务端响应签名 SDK 验签
海量心跳压力在线服务独立扩缩容 + Redis + 间隔可调 + 限流

前后端落点：后端（激活/心跳/CRL/席位服务）；前端（控制台监控在线实例/席位/心跳异常）。

步骤 10：防破解专项

① 设计思路 聚焦三类攻击：伪造、共享、绕过。伪造→签名；共享→指纹/席位/水印；绕过→Native+完整性+时间回拨检测。

② 技术选型对比 / ③ 风险 / ④ 应对（合并）

攻击防护（应对）
伪造 License非对称签名，私钥不出域
一码多机离线水印/软节点锁 + 在线席位去重
时间回拨绕有效期在线同步可信时间 + 本地 highWaterMark（见过的最大时间戳，回拨即拒）+ 单调用量计数
patch 验签Native 化 + 多点冗余 + 完整性自校验 + 解密密钥依赖 License
改本地缓存缓存加密+签名，以服务端为准

前后端落点：后端 + SDK；前端无关。

步骤 11：License 生命周期、版本号与历史追溯

① 设计思路 状态机：DRAFT→ISSUED→ACTIVATED→(RENEWED/MODIFIED ↺, SUSPENDED↔)→REVOKED/EXPIRED。每次签发/续期/变更/吊销 licenseVersion+1 并写 license_history（完整 claims 快照 + 签名 + 操作人 + 时间 + 原因 + 订单）。控制台支持版本列表、任意两版 diff、当前生效标识、吊销原因、审计。到期前 30/15/7 天多渠道预警。

② 技术选型对比

能力选型备选
历史存储不可变历史表（append-only）事件溯源 ES
diff 展示前端 JSON diff 组件后端预算 diff
预警定时任务 + 站内/邮件/短信消息队列

③ 潜在风险 / ④ 应对措施

风险应对
运营误吊销/误改二次确认 + 审批流 + 历史可回看回溯
历史被篡改append-only + 审计 + 可选哈希链
到期未续致中断预警 + 续期宽限 + 平滑降级

前后端落点：后端（生命周期 API + 历史/审计）；前端（生命周期操作、版本历史时间线 + diff 视图）。

步骤 12：客户端 License SDK

① 设计思路 轻量 Java 库嵌入被保护应用：指纹/软指纹采集、公钥验签、离线 .lic 加载/在线快照缓存（加密）、激活/心跳、时间回拨检测、功能/版本门禁（注解 + 编程式）、事件回调（将到期/已降级/已吊销）。抽象 LicenseProvider 接口，便于切换自研/Keygen/SaaS。

② 技术选型对比

维度选型备选
门禁@RequiresFeature + Spring AOP拦截器
关键逻辑JNI native（步骤 3）纯 Java（弱）
提供方抽象LicenseProvider SPI硬编码

③ 潜在风险 / ④ 应对措施

风险应对
SDK 被逆向定位校验关键逻辑 Native + 多点冗余
误降级影响业务宽限期 + 平滑降级（停增量保只读）+ 状态端点供运维排查
集成侵入大注解式低侵入 + starter 自动装配

前后端落点：后端（嵌入客户应用）；客户应用前端可读 LicenseStatus 展示授权状态/到期/席位。

第三部分 · 多租户运营平台（前后端）

步骤 13：多租户隔离与租户上下文

① 设计思路 平台自身用共享库 + tenant_id 行级隔离，PG RLS 行级安全做数据库兜底（应用层漏判也拒绝跨租户）。被授权客户业务系统多为私有化独立部署。平台支持可配置隔离级别：大客户独立 Schema/库，中小走共享行级。TenantContext（ThreadLocal）+ 拦截器解析 JWT 注入 + MyBatis-Plus 多租户插件自动追加条件。

② 技术选型对比

策略隔离成本弹性适用
独立库最高高差强合规/大客户
独立 Schema中高中中数十~数百
共享+tenant_id+RLS ⭐逻辑+DB兜底低最好海量中小

③ 潜在风险 / ④ 应对措施

风险应对
跨租户越权应用拦截 + PG RLS 双保险 + 跨租户安全测试
忘加 tenant 条件插件全局拦截 + RLS 兜底 + 代码评审

前后端落点：后端（隔离/上下文/RLS）；前端（登录态携带租户、超管可切租户视图）。

步骤 14：租户开通、DB 初始化与超管下发

① 设计思路 用 Saga 状态机 编排：建租户记录(INITIALIZING)→按隔离级别编排资源→Flyway 迁移建表→种子数据→创建初始超管→绑套餐生成首张 License→置 ACTIVE。每步幂等可重试，失败回滚/补偿，杜绝半初始化脏数据。超管用邀请激活链接自设密（优于系统发密码），首登强制改密 + 强制 MFA，密码 Argon2/BCrypt 加盐，绝不明文落库/日志。

② 技术选型对比

环节选型备选
编排Saga 状态机本地事务（跨资源不够）
DB 迁移FlywayLiquibase
密码哈希Argon2idBCrypt
超管下发邀请链接(一次性限时)系统生成密码
MFATOTP短信验证码

③ 潜在风险 / ④ 应对措施

风险应对
半初始化脏数据Saga + 幂等 + 回滚补偿 + 断点续做
超管密码泄露邀请自设密 + 首登改密 + MFA + 不落明文 + 审计
存量租户 schema 升级风险Flyway 灰度 + 备份 + 回滚演练 + 维护窗口

步骤 15：授权模型（产品-模块-功能-版本-套餐-订阅）
