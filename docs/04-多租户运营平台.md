# 04 · 多租户运营平台（前后端）

> 目标：提供租户开通、DB 与初始数据自动化初始化、初始超管账号下发；以"产品—模块—功能—版本"为核心的可运营授权模型，驱动 License 签发。含后端服务与前端控制台。

---

## 1. 租户隔离策略选型

| 策略 | 隔离级别 | 成本 | 弹性 | 适用 |
|------|----------|------|------|------|
| **独立数据库**（DB per tenant） | 最高（物理隔离） | 高（实例多、运维重） | 差（千租户难管） | 强合规/大客户/私有化 |
| **共享库独立 Schema**（Schema per tenant） | 中高 | 中 | 中 | 数十~数百租户 |
| **共享库共享表 + tenant_id**（Discriminator/行级） | 逻辑隔离 | 低 | **最好** | 海量中小租户 SaaS |

> **结论（推荐分层策略）**：
> - **运营平台自身**：共享库 + `tenant_id` 行级隔离（PostgreSQL **RLS 行级安全**兜底防越权）。
> - **被授权客户的业务系统**：多为**私有化独立部署 / 独立库**（每个客户独立环境）。
> - 平台支持**可配置隔离级别**：大客户可选独立 Schema/独立库，中小客户走共享行级，兼顾成本与合规。

### 1.1 行级隔离实现
- 全局 `TenantContext`（ThreadLocal / Reactor Context）保存当前租户。
- 登录/请求入口的拦截器解析 JWT 中 `tenantId` 注入上下文。
- MyBatis-Plus **多租户插件**自动给 SQL 追加 `tenant_id = ?`。
- **PG RLS** 作为数据库层兜底：即使应用层漏判，数据库也拒绝跨租户读取（双保险）。

## 2. 租户开通与初始化流程

```
运营/客户自助下单
   │
   ▼
① 创建租户记录(tenant: 状态=INITIALIZING)
   │
   ▼
② 资源编排（按隔离级别）
   ├─ 共享行级：无需建库，仅写 tenant 记录
   ├─ 独立 Schema：CREATE SCHEMA t_xxx
   └─ 独立库：申请实例/库
   │
   ▼
③ 执行 Flyway 迁移（版本化建表）—— 幂等、可重试
   │
   ▼
④ 种子数据 seed（字典、默认角色、菜单、默认配置）
   │
   ▼
⑤ 创建初始超管账号
   ├─ 用户名规则 + 随机强密码（或邀请链接）
   ├─ 密码 BCrypt/Argon2 加盐存储
   ├─ 标记 mustChangePassword=true（首登强制改密）
   └─ 安全下发：邮件/短信一次性链接（不明文落库/日志）
   │
   ▼
⑥ 绑定套餐/授权 → 生成首张 License（调用 03 文档签发）
   │
   ▼
⑦ 租户状态=ACTIVE，发送开通通知
   │
   ▼
失败任一步 → 回滚/补偿（Saga），状态=FAILED，可重试
```

### 2.1 关键设计
- **幂等与可重试**：开通是分布式动作，用**Saga/状态机**编排，每步幂等，失败可断点续做，避免半初始化租户。
- **DB 初始化**：Flyway 脚本纳入版本控制（`V1__init.sql`、`V2__add_xxx.sql`），新租户始终建到最新 schema；存量租户随平台升级统一迁移。
- **超管账号安全**：
  - 绝不在数据库/日志中存明文密码；
  - 推荐**邀请激活链接**（一次性、限时）由客户本人设密，优于系统生成密码；
  - 首登强制改密 + 强制绑定 MFA（TOTP）；
  - 超管操作全程审计。
- **租户配额**：开通即写入资源配额（用户数、存储、QPS），与 License `features` 一致。

## 3. 授权模型：产品—模块—功能—版本

```
Product(产品)
  └── Module(模块)            例：风控、报表、审计
        └── Feature(功能点)    例：REPORT.EXPORT、RISK.REALTIME
        └── Quota(配额项)      例：MAX_USERS、MAX_QPS
Version(版本)                  例：v2.0、v2.1（功能随版本演进）
Plan/SKU(套餐)                 = 模块/功能/配额/版本范围的组合 + 价格
Subscription(订阅)            租户 ↔ 套餐 的购买关系（含周期、数量）
License                        由 Subscription 生成（03 文档），下发给客户
```

- **功能矩阵**：`Feature × Version` 决定"某功能在某版本是否存在/可用"，支持**按版本授权**（如 EXPORT 仅 v2.1+ 提供）。
- **套餐编排**：运营在控制台用"勾选模块/功能/配额 + 选版本范围 + 定价"组装 SKU，支持基础版/专业版/企业版分层与加购包。
- **授权 → License**：订阅生效后，系统把套餐解析为 License 的 `modules/features/appVersionRange/concurrency`，调用签发引擎生成 License。变更套餐 → 重签 License（版本+1，见 03 文档 §7）。

## 4. 数据模型（核心表 ER）

```
tenant(租户)
  id, code, name, isolation_level, db_ref, status, plan_id, created_at ...

tenant_admin / sys_user(用户)
  id, tenant_id, username, password_hash, must_change_pwd, mfa_secret, status, role_id ...

sys_role / sys_permission / role_permission   —— RBAC

product / module / feature / app_version
  feature(id, module_id, code, name)
  app_version(id, product_id, version, released_at)
  feature_version_map(feature_id, version_id, available)   —— 版本功能矩阵

plan(套餐)                id, product_id, name, price, version_range
plan_item(套餐项)         plan_id, feature_id | quota_key, quota_value

subscription(订阅)        id, tenant_id, plan_id, qty, start_at, end_at, status

license(授权)             id, license_id, tenant_id, subscription_id,
                          current_version, mode, machine_code, status,
                          not_before, not_after, watermark ...
license_history(历史)     id, license_id, version, claims_json, signature,
                          operator, op_type, reason, order_ref, created_at  ——【历史追溯】

activation(在线激活)      id, license_id, instance_id, machine_code, last_heartbeat, status
crl(吊销列表)             license_id, reason, revoked_at
usage_metric(用量)        license_id, metric, value, period
audit_log(审计)           actor, tenant_id, action, target, ip, detail, created_at
order(订单)/contract(合同)（可选，对接计费）
```

> `license_history` 是**历史追溯**的核心：每次变更保存完整 claims 快照与签名，支持版本回看与 diff；`audit_log` 记录所有运营动作。

## 5. 后端架构

```
license-platform (Spring Boot 3.x)
├── module-auth        运营后台登录、RBAC、JWT、MFA
├── module-tenant      租户开通/初始化/Saga 编排/配额
├── module-catalog     产品/模块/功能/版本/功能矩阵
├── module-plan        套餐/订阅/计费对接
├── module-license     签发引擎(调 KMS 签名)/生命周期/版本历史/CRL
├── module-online      激活/心跳/并发席位(Redis)/可信时间/响应签名
├── module-stats       用量计量、统计大盘数据
├── module-audit       审计日志(不可变)
├── common-tenant      TenantContext + MyBatis 多租户拦截 + PG RLS
└── infra              KMS/HSM 适配、消息(到期预警)、缓存、定时任务
```

- **签发引擎**：组装 claims → 调 KMS/HSM 签名（私钥不出域）→ 落 `license`+`license_history` → 产出 `.lic`/在线快照。
- **在线服务**：高并发心跳，Redis 维护活跃实例与席位，CRL 缓存；与签发分离部署，可独立扩缩容。
- **定时任务**：到期扫描预警、过期置态、用量归集、长期未心跳实例回收席位。
- **安全**：运营后台接口 RBAC + 操作审计；SDK 通道 mTLS；敏感配置加密。

### 5.1 关键 API（示意）
```
# 运营后台（前端调用，JWT + RBAC）
POST /api/tenants                 开通租户
POST /api/plans                   创建套餐
POST /api/subscriptions           创建订阅
POST /api/licenses/issue          签发 License
POST /api/licenses/{id}/renew     续期（版本+1）
POST /api/licenses/{id}/modify    变更（版本+1）
POST /api/licenses/{id}/revoke    吊销（入 CRL）
GET  /api/licenses/{id}/history   历史版本列表 + diff
GET  /api/stats/dashboard         运营大盘

# SDK 通道（客户端调用，mTLS + 签名）
POST /sdk/activate                激活
POST /sdk/heartbeat               心跳（返回签名快照）
GET  /sdk/crl                     吊销检查
GET  /sdk/time                    可信时间
```

## 6. 前端架构（运营管理控制台）

```
license-console (Vue3 + TS + Vite + Pinia + Element Plus / Ant Design Vue)
├── 登录/权限       账号密码 + MFA，按钮级权限(RBAC)
├── 租户管理        开通向导、列表、状态、配额、初始化进度
├── 产品目录        产品/模块/功能/版本、版本功能矩阵编辑
├── 套餐管理        SKU 编排（勾选模块/功能/配额+版本+定价）
├── 订阅管理        下单/续费/变更/退订
├── License 管理    签发向导、列表、详情
│     ├── 生命周期   续期/变更/吊销/暂停
│     ├── 版本历史   版本列表 + 任意两版 diff（高亮变化）   ←【历史追溯】
│     └── 下载/重发  .lic 文件、激活码
├── 在线监控        激活实例、在线席位、心跳异常、地理分布
├── 运营大盘        ECharts：授权数/到期分布/用量趋势/收入
└── 审计中心        操作审计查询、导出
```

- **开通向导**：分步表单（租户信息→隔离级别→套餐→超管设置→确认），实时展示初始化进度（轮询/WebSocket）。
- **历史追溯页**：时间线 + 版本快照 + diff 视图，满足"用户进行管理及历史追溯，包括版本号"。
- **国际化/主题**：i18n + 暗色主题，适配政企信创环境（可换国产浏览器/麒麟）。

## 7. 潜在风险与应对

| 风险 | 应对 |
|------|------|
| 租户间数据越权 | 应用层多租户拦截 + 数据库 **PG RLS** 双保险；安全测试覆盖跨租户用例 |
| 半初始化租户（脏数据） | Saga 状态机 + 每步幂等 + 失败回滚/补偿 + 重试 |
| 超管密码泄露 | 邀请链接自设密 + 首登强制改密 + MFA + 不落明文/日志 + 审计 |
| 存量租户 schema 升级风险 | Flyway 灰度迁移 + 备份 + 演练回滚 + 维护窗口 |
| 授权与 License 不一致 | 套餐→License 单一映射来源，变更即重签；定时对账 |
| 海量心跳压力 | 在线服务独立扩缩容 + Redis + 心跳间隔可调 + 限流 |
| 计费/订单对接复杂 | 订阅与计费解耦，预留接口对接现有计费/合同系统 |
| 运营误操作（错误吊销/变更） | 二次确认 + 操作审计 + 可回看历史版本 + 关键操作审批流 |
| 私有化客户无外网 | 控制台支持**离线签发**导出 .lic，纯离线运营子集 |
| 大客户要求物理隔离 | 可配置隔离级别（独立 Schema/库），按客户分级 |

## 8. 与前两份文档的衔接

- 平台**签发**的 License 即 03 文档定义的签名结构；
- 客户拿到的被保护交付物由 02 文档的流水线产出，内置 License SDK；
- 平台的"产品/模块/功能/版本"模型直接映射到 License 的 `modules/features/appVersionRange`，实现"配置即授权、授权即生效"。
