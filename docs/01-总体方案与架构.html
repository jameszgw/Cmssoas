⚠️ 诚实前提：Java 字节码本质可逆，世界上不存在"绝对无法破解"的纯软件保护方案。本方案的目标是抬高破解成本到超过软件价值，并通过 License + 在线吊销 + 法律手段（LICENSE/合同）构建综合防线。所有"防护"均以"提高门槛 + 可发现 + 可追责"为现实目标，文中不会承诺"无法被破解"。

1.2 端到端核心流程（概览）

离线授权流程

  1. 客户在目标机器运行"指纹采集工具"（SDK 提供），生成机器指纹 machineCode。
  2. 客户将 machineCode + 订单号提交给运营方。
  3. 运营人员在控制台选择租户/套餐/版本/有效期 → 后端用私钥签名生成 .lic 文件（绑定该指纹）。
  4. 客户导入 .lic 到应用 → SDK 用内置公钥验签 + 校验指纹/有效期/版本/功能 → 放行。

在线授权流程

  1. 应用启动时携带 License 凭证（或激活码）向 License Server 注册/激活。
  2. SDK 定期发送心跳（带 nonce 防重放、mTLS）→ 服务端校验有效性、并发席位、是否被吊销 → 返回服务端签名的最新授权快照。
  3. 网络抖动时启用宽限期（grace period）离线容忍；超期或被吊销则降级/停服。

2. 设计原则

  1. 纵深防御（Defense in Depth）：任何单点都可能被攻破，故"混淆 + 加密 + Native + License + 在线吊销 + 审计"层层叠加，攻击者需要逐层突破。
  2. 非对称信任根：License 用厂商私钥签名、客户端用公钥验签。客户端永不持有私钥，因此即使被完全逆向，也无法伪造合法 License（只能尝试 patch 校验逻辑，而这正是代码保护层要拦截的）。
  3. 失败策略分级：
    • 验签失败 / 指纹不匹配 / 已过期 / 被吊销 → 拒绝启动或核心功能降级（fail-closed）。
    • 在线校验网络异常 → 进入宽限期（fail-open 有限期），避免误伤生产。
  4. 可观测、可追溯、可追责：所有签发、变更、吊销、激活、心跳异常均落审计日志，支撑历史追溯与法务取证。
  5. 运营友好：授权模型与计费/套餐解耦但联动，支持自助开通、批量签发、到期预警、续费、灰度升级。
  6. 密钥分级与轮换：按产品线/大版本使用不同签名密钥；私钥托管 KMS/HSM，支持轮换与吊销。
