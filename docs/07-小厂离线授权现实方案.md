# 07 · 小厂离线授权现实方案（替代硬件指纹绑定）

> 针对反馈："离线靠硬件指纹绑定 `.lic`，对小厂不现实。"
> 本文给出**为什么不现实**，以及一套**按投入从低到高、可落地**的替代方案，供中小团队按软件价值与客户环境分级选用。

---

## 1. 为什么"硬件指纹绑定"对小厂不现实

| 痛点 | 说明 |
|------|------|
| **人工流程重** | "采集指纹 → 客户回传 → 人工签发 → 回传 → 客户导入" 每客户/每机器/每次硬件变更都要走一遍。小厂没有专职 License 运维，签发即人肉客服。 |
| **云/容器无稳定指纹** | 自动扩缩容、镜像复制、虚机迁移、K8s 重调度都会让硬件 ID 漂移，授权频繁失效。现代部署越来越"无硬件"。 |
| **跨平台 native 维护贵** | 指纹采集要覆盖 Windows / 各家 Linux / 麒麟 / 统信，native 编译与适配对小团队是持续负担。 |
| **客户体验差、抵触锁机** | B2B 客户讨厌被"绑死在某台机器"，迁移困难、扩容受阻，直接影响满意度与续费。 |
| **支持成本爆炸** | 换块硬盘、加块网卡就失效 → 紧急工单淹没小团队，得不偿失。 |

> **结论**：硬件指纹是"高价值软件 + 涉密离线"才值得的重投入。小厂应优先选下面**更轻**的方案，把硬绑定作为可选项而非默认。

---

## 2. 替代方案总览（决策矩阵）

| 方案 | 是否绑机器 | 是否需联网 | 厂商运维成本 | 防复制能力 | 适用 |
|------|-----------|-----------|------------|-----------|------|
| **T0 签名期限 License（不绑机）** | 否 | 否 | 极低 | 弱（靠水印+合同+遥测） | 客户少、信任高、按客户/项目计价、涉密内网 |
| **T1 软节点锁（installationId）** | 软绑（随机ID，非硬件） | 否（可选联网去重） | 低 | 中（防随手复制） | 想要"一份一机"但怕硬件运维 |
| **T2 在线自助激活 + 离线运行** ⭐ | 软绑 | 仅激活/续租时一次 | 低（自动化，无人工） | 强（在线去重/限激活数/可吊销） | 客户有外网，平时可离线运行 |
| **T3 用现成/开源自托管 Licensing** ⭐ | 取决于配置 | 取决于配置 | 低（不自研） | 强 | 想几天上线、不想自研签发/激活/后台 |

> ⭐ 推荐小厂默认从 **T0** 起步，需要防复制时上 **T2** 或直接 **T3**。

---

## 3. T0 · 签名期限 License（不绑机器，最常用）

**思路**：License 只签 `客户 + 模块 + 功能 + 版本范围 + 有效期 + 唯一水印`，**不绑定任何机器**。
防伪造靠**非对称签名**（私钥不出域）；防滥用靠**有效期（到期必续）+ 水印溯源 + 合同 + 可选遥测**。

```jsonc
{
  "licenseId":"LIC-...", "licenseVersion":1,
  "customer":"XX公司", "productCode":"CODEMAN", "edition":"PRO",
  "modules":["RISK","REPORT"], "features":{"REPORT.EXPORT":true,"MAX_USERS":100},
  "appVersionRange":">=2.0.0 <3.0.0",
  "notBefore":"2026-06-06", "notAfter":"2027-06-06",
  "watermark":"WM-7f3a..."        // 客户唯一水印，泄露可溯源
  // 注意：无 machineCode
}
```

- **签发零人工**：可在控制台一键生成，甚至随订单自动生成 `.lic` 邮件发出。
- **客户体验最好**：随便部署、随便迁移、云容器无感。
- **到期续签**即天然的"再次授权"节点，是小厂主要的商业抓手。

| 风险 | 应对 |
|------|------|
| 一份 License 被复制到多处部署 | ①唯一**水印**：泄露包可溯源到客户，配合合同追责；②**有效期**强制周期性回到厂商续费；③可选**轻量遥测**：应用偶尔上报部署数/客户标识，后台发现异常多副本即预警 |
| 无法实时停用 | 靠到期不续 + 法律手段；如需实时吊销，升级到 T2 |

> 这是**绝大多数小厂 B2B/私有化的现实选择**：成本最低、客户最舒服，用商业+法律+水印兜住技术上的"可复制"。

## 4. T1 · 软节点锁（installationId，不碰硬件）

**思路**：首次启动随机生成 `installationId`（UUID），加密存到本地数据目录/配置；激活时把 License **绑定 installationId** 而非硬件。

```
首次启动 → 生成 installationId(随机) → 存本地(加密)
签发/激活 → License 绑定该 installationId
启动校验 → 验签 ∧ 本地 installationId == License 中绑定值 ∧ 有效期
换机/迁移 → 客户自助“反激活”释放 → 新环境重新激活(限次数)
```

- 比硬件指纹**友好得多**：云容器只要**持久化一个文件**即可稳定；无需 native 采集、无跨平台编译。
- 适合"想要一份授权对应一个安装实例、但不愿碰硬件运维"的场景。

| 风险 | 应对 |
|------|------|
| 复制整个安装目录 → 复制 installationId | 纯离线无法根治；配合**水印 + 有效期**；如可联网，用 T2 的"在线去重"（同一 installationId 多地并发上报即告警/拒绝） |
| 客户误删数据目录致 ID 丢失 | 自助重新激活（限次）+ 备份提示 |

---

## 5. T2 · 在线自助激活 + 离线运行 ⭐（强烈推荐）

**核心：把"人工签发"变成"自动激活服务"，干掉小厂最大的人力痛点；激活后可离线运行。**

```
厂商发放 license key（一串激活码，对应一笔订单/套餐）
        │
客户在应用中填入 license key
        │  ① 联网一次：POST /activate { licenseKey, installationId }
        ▼
轻量激活服务（一台小 VPS / Serverless 即可）
   - 校验 key 有效 & 未超激活数（如 1 key 限 3 次激活）
   - 绑定 installationId，自动用私钥签发 License 快照
        │  ② 返回签名 License（缓存本地）
        ▼
客户应用：之后【离线运行】，凭本地缓存的签名 License 工作
        │
（可选）每隔 N 天联网“续租”一次刷新；长期不联网 → 宽限期后降级
换机：客户自助“反激活”释放名额 → 新机重新激活
```

**为什么适合小厂**
- **无人工**：激活/签发/换机全自动，客户自助，厂商零介入。
- **轻量**：一台便宜 VPS 或云函数 + 一个小数据库即可承载激活，无需常驻心跳。
- **防复制强**：在线限激活次数 + 同一 key 去重 + 可远程吊销。
- **离线运行**：激活后断网也能跑（适合客户内网/不稳定网络），只在激活/续租时短暂联网。

**与在线心跳模式的区别**：T2 不要求持续心跳，只在"激活/续租"时联网，介于纯离线与纯在线之间，运维与体验平衡最好。

| 风险 | 应对 |
|------|------|
| 激活服务宕机致客户无法激活 | 服务无状态 + 便宜多实例；激活是低频操作，容忍短暂排队；保留 T0 离线签发兜底 |
| 客户完全无外网 | 提供"离线激活码"通道：客户导出 installationId 文本，厂商网页生成离线 License（半自动），即退化为 T1/T0 |
| 滥用激活名额 | 每 key 限激活次数 + 反激活回收 + 异常并发告警 |
| 续租期网络不稳 | 宽限期（如 72h~30d）内继续运行，不误伤生产 |

## 6. T3 · 直接用现成 / 开源自托管 Licensing（降本利器）

小厂**不必自研**签发引擎、激活服务、吊销、管理后台——这些自研工作量巨大。可直接用成熟方案：

| 方案 | 形态 | 特点 | 适合 |
|------|------|------|------|
| **Keygen** (keygen.sh) | 开源，可**自托管**，也有云版 | License/激活/机器管理/吊销/分发 API 齐全，REST 易集成 Java | 想自控数据又不想自研 ⭐ |
| Cryptolens | SaaS + SDK | 成熟、含离线激活、防篡改 | 快速上线、可接受 SaaS |
| LicenseSpring | SaaS | 功能全、管理后台现成 | 预算充足、要省事 |
| 自建（本仓库 01–06 方案） | 自研 | 完全可控、可深度定制 | 软件价值高、长期投入 |

- **建议**：小厂起步用 **Keygen 自托管**（数据自控、合规友好）或 SaaS，几天集成上线；待规模/价值上来再考虑自研（迁移到 01–06 方案）。
- **风险/应对**：
  - 依赖第三方/数据出境 → 选**可自托管开源（Keygen）**规避；
  - 订阅成本 → 早期成本远低于自研人力；
  - 锁定 → 抽象一层 `LicenseProvider` 接口，便于未来切换到自研。

## 7. 给小厂的推荐决策路径

```
客户少 / 信任高 / 涉密内网 / 按客户计价
        └──▶ T0 签名期限 License（不绑机）+ 水印 + 合同        【首选，零运维】

需要“防随手复制”但客户有外网
        └──▶ T2 在线自助激活 + 离线运行                       【推荐，自动化无人工】

不想自研签发/激活/后台，想几天上线
        └──▶ T3 Keygen 自托管 / SaaS                          【最省事】

软件价值很高 / 强涉密离线 / 愿意重投入
        └──▶ 才考虑硬件指纹绑定（01–06 完整方案的可选增强）     【最后才上】
```

**一句话**：小厂离线授权应以 **T0（签名期限，不绑机）** 为默认，需要防复制时上 **T2（自助激活）** 或 **T3（现成方案）**；硬件指纹是高价值/涉密场景的可选增强，**不要默认上**。

---

## 8. 通用底线（无论选哪个 Tier）

- **永远用非对称签名**：私钥不出域，客户端只验签，杜绝伪造（成本几乎为零，必做）。
- **永远打唯一水印**：泄露可溯源 + 合同追责，是最便宜的"防复制"。
- **永远设有效期**：到期续签是商业抓手，也是技术上的周期性"重新授权"。
- **失败策略**：离线/网络异常给**宽限期**，宁可短期放行也别误伤客户生产。
